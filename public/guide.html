Go Live / Stop

  • Go Live — click the "Go Live" button on a stream card. Status changes to Live and health simulation begins (bitrate, fps, viewers update every 2 seconds).

SRT / SRTLA Bonding & Latency

SRTLA (Secure Reliable Transport Link Aggregation) bonds multiple internet connections into one reliable stream. It's ideal for mobile/IRL streaming where a single connection may be unstable.

Bond Modes Explained

Aggregate
Combine bandwidth
All interfaces work together. Throughput = sum of all connections. Packet loss on one is compensated by others. Best for high-bitrate stable streaming.
Broadcast
Duplicate for redundancy
Same data sent over every interface. Receiver picks the first copy that arrives. Highest reliability, consumes more bandwidth. Best for mission-critical streams.
Failover
Auto-switch on failure
One primary interface active. If it drops, a secondary takes over seamlessly. Best for cellular + WiFi backup scenarios.

Interface Monitoring

Each bond has simulated interfaces (Cellular 5G, WiFi 6, Ethernet). Live stats update every 2 seconds showing:

  • Status — Active / Degraded
  • Throughput — Mbps per interface
  • Latency — round-trip time in ms
  • Packet Loss — percentage of lost packets
  • Jitter — variance in packet arrival time (ms)
  • Packets Sent / Received — cumulative counts
  • Total Throughput — combined from all interfaces

Tip: In Aggregate mode, a 5G interface (30 Mbps) + WiFi 6 (20 Mbps) gives you ~50 Mbps total throughput — enough for 4K streaming.

SRT Latency Presets Reference

SRT latency controls the buffer size for error correction. Lower latency = less delay but less tolerance for network issues. Higher latency = more delay but better reliability.

PresetValueUse CaseNetwork Requirement
Ultra Low50msLAN / wired studio<1ms jitter, 0% packet loss
Low60msStable WiFi / local<5ms jitter, <0.1% loss
Standard90msRecommended for most<10ms jitter, <0.5% loss
Balanced120msDefault / safe choice<20ms jitter, <1% loss
High Reliability200msCellular / 4G<50ms jitter, <2% loss
Max Buffer500msUnstable connections<100ms jitter, <5% loss
Custom20–8000msManual tuningVaries based on value

Choosing the Right Latency

  • 50ms–60ms — Only use if you have a wired studio setup with guaranteed bandwidth. Great for two-way communication (interviews, esports commentary).
  • 90ms — The sweet spot for most streaming. Low enough for real-time interaction, high enough to handle minor network glitches.
  • 120ms — The default SRT latency. Reliable for WiFi streaming. Good balance of delay vs. stability.
  • 200ms–500ms — For cellular bonding or unreliable networks. Higher latency gives the SRT protocol more time to retransmit lost packets.
  • Custom — Use when you know your network's exact characteristics. Set it to match your worst-case RTT + jitter * 3.

Formula: SRT Latency ≥ (Round Trip Time × 2) + (Jitter × 3). For example, if your RTT is 30ms and jitter is 10ms: minimum latency = 60 + 30 = 90ms.

Frequently Asked Questions

What is SRT?

SRT (Secure Reliable Transport) is an open-source video transport protocol that delivers high-quality video over unpredictable networks. It uses packet retransmission, end-to-end encryption, and adaptive bitrate to maintain stream quality even with packet loss up to 50%.

What is SRTLA?

SRTLA (SRT Link Aggregation) bonds multiple internet connections together. Instead of relying on a single network, SRTLA uses all available connections simultaneously (Aggregate), duplicates data for redundancy (Broadcast), or switches on failure (Failover).

How do I connect OBS to StreamCast?

After creating a stream, click "Copy Key" to get the ingest URL and stream key. In OBS: Settings → Stream → Service = Custom → Server = [ingest URL] → Stream Key = [your key]. For SRT, use the SRT URL format: srt://localhost:3000?streamid=your-stream-id.

Why are viewer numbers simulated?

This is a demo platform. In production, viewer counts would come from CDN analytics (CloudFront, Mux, etc.) or from each destination platform's API. The simulation shows how the dashboard would look with real data.

How does multistream work?

Multistream takes your single video feed and sends it to multiple platforms simultaneously. StreamCast would transcode the video once and then forward copies to YouTube, Twitch, Facebook, etc. This saves bandwidth on your upload since you only send one stream to StreamCast.

What is the difference between RTMP and SRT?

  • RTMP — older protocol (Adobe Flash era). Low latency but no error correction. Breaks easily with packet loss.
  • SRT — modern protocol. Built-in encryption, error correction, adaptive to network conditions. Works over any network (including cellular).

Can I use my own domain?

This demo runs on localhost. In production, StreamCast would be deployed to a VPS or cloud provider with a custom domain and SSL certificate.

How is my data protected?

  • Passwords are hashed with bcrypt (10 salt rounds)
  • API calls use JWT tokens (expire after 7 days)
  • SRT streams can be encrypted with AES-128 or AES-256
  • All connections go through your local server

What if I forget my password?

The demo has no password reset flow. Since this is a local demo, you can create a new account or use the demo credentials. In production, a "Forgot Password" flow with email reset would be implemented.

What is the technology stack?

Backend: Node.js, Express, Socket.IO, sql.js (SQLite), bcryptjs, JWT. Frontend: Vanilla JS SPA, HLS.js, Canvas API (no chart libraries). Styling: Custom CSS with dark theme, glassmorphism, Inter font.
